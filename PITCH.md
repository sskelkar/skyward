# SKYWARD: Why We Built This Game

## The Problem

**The future of software development isn't writing code—it's orchestrating agents that write code.**

Most engineers today use AI assistants like Claude as glorified autocomplete: ask a question, get an answer, copy-paste code. But the real productivity breakthrough comes from **multi-agent orchestration**—coordinating multiple AI agents working in parallel, each with specialized roles, building complex systems faster than any human could alone.

**The gap:** IC engineers understand single-agent workflows, but lack hands-on experience with:
- Coordinating multiple agents simultaneously
- Using agent personas to drive better technical decisions
- Building subagent hierarchies and agent teams
- Managing the complexity of multi-agent architectures

**The consequence:** As AI coding tools become ubiquitous, engineers who only know single-agent patterns will be left behind. Those who master multi-agent orchestration will 10x their output.

---

## The Solution: Learn By Building Something Real

**Skyward** is a half-day workshop where engineers learn multi-agent orchestration by doing it—not by watching slides or following tutorials, but by building an actual airline startup's engineering systems in 4 hours.

### Why Game-Based Learning Works

**Traditional training fails because:**
- ❌ Slides don't stick—people forget theory without practice
- ❌ Toy examples feel artificial—no pressure, no real decisions
- ❌ No feedback loop—you don't know if you're doing it right

**Game-based learning succeeds because:**
- ✅ **Immersion:** Role-playing as directors creates real emotional stakes
- ✅ **Pressure:** Quarterly demos force you to deliver, not just plan
- ✅ **Iteration:** Four quarters = four chances to learn from mistakes
- ✅ **Peer learning:** See how others orchestrate agents differently
- ✅ **Safe failure:** Mistakes teach without real-world consequences

### Why the Director Role-Play Matters

Participants aren't just "developers with AI tools"—they're **directors of engineering** commanding a department. This reframe is crucial:

**As a developer:**
- You think: "I need to write this code"
- You use Claude: "Write me a login function"

**As a director:**
- You think: "My department needs to deliver a booking system"
- You orchestrate agents: "Agent 1: Build the search API. Agent 2: Build the frontend. Agent 3: Integrate them."

This mindset shift—from implementer to orchestrator—is exactly what the AI-augmented future requires.

## What Participants Actually Learn

### Primary Learning Objectives (Multi-Agent Orchestration)

**1. Parallel Agent Execution**
- Run multiple agents simultaneously on independent tasks
- Understand when parallel vs sequential execution makes sense
- Practice managing multiple agent outputs at once

**Example from the game:**
```
Q2 (60 minutes to build for investor demo):
- Agent Team A: Build backend API (runs in parallel)
- Agent Team B: Build frontend UI (runs in parallel)
- Agent Team C: Integration & polish (runs after A+B complete)
```

**2. Agent Personas and Role-Based Prompting**
- Use specialized agent roles (PM, architect, QA, frontend dev, backend dev)
- Understand how persona affects decision quality
- Learn when to use which type of agent

**Example from the game:**
```
Q1 (Research & Vision):
- Business Analyst agent: Research competitors
- Product Manager agent: Define requirements
- UX Designer agent: Create wireframes
- Technical Architect agent: Design service architecture
```

**3. Subagents and Hierarchical Teams**
- Build agent hierarchies (director → team leads → specialists)
- Delegate to subagents effectively
- Manage context and handoffs between agent layers

**Example from the game:**
```
Director delegates to Department Head agent:
  ↳ Department Head spawns Service Team agents:
      ↳ Service Team A: Search API
      ↳ Service Team B: Booking API
      ↳ Service Team C: Frontend
```

**4. Iterative Refinement Patterns**
- Build → Review → Improve cycles with different agents
- Use specialist agents to refine generalist work
- Balance speed vs quality with agent iteration

**Example from the game:**
```
Q3 (Polish phase):
- Builder agent: Add new feature
- QA agent: Review and find issues
- Refiner agent: Fix issues and improve UX
```

**5. Context Management at Scale**
- Handle multiple codebases across services
- Manage agent context windows effectively
- Use integration agents to bridge services

**6. Real-World Agent Tradeoffs**
- Speed vs quality (ship fast vs iterate for polish)
- Cost vs capability (when to use expensive models)
- Specialization vs generalization (one agent for everything vs team of specialists)

---

### Bonus Learning Objectives (Management Glimpse)

While the primary goal is teaching multi-agent orchestration, participants also gain unexpected insights into engineering management:

**1. Operating Under Constraints**
- **Time pressure:** Quarterly demos force prioritization
- **Quality vs speed:** Sometimes "good enough" beats "perfect but late"
- **Tech debt awareness:** Quick wins in Q2 may cost you in Q4
- **Resource efficiency:** API costs matter—can't brute-force everything

**2. Business Context**
- **Why before what:** Understanding business goals before building features
- **User impact:** Directors think about customers, not just code
- **Department goals:** How your work fits into company success
- **Strategic thinking:** What to build first, what to defer

**3. Delegation and Trust**
- **Let go of implementation:** Directors set direction, agents execute
- **Review not rewrite:** Check quality without micromanaging
- **Clear requirements:** Vague instructions produce vague results
- **Feedback loops:** How to improve team output iteratively

**4. Cross-Functional Collaboration**
- **API contracts:** How departments integrate without blocking each other
- **Dependencies:** Understanding upstream/downstream relationships
- **Communication:** Demos require explaining technical work to non-technical stakeholders (investors)

## The Pedagogical Philosophy

### Learning Through Role-Play

**Why "Director of Engineering" instead of "Developer"?**

Because the role shapes the thinking:

- **Developers** think about implementation details
- **Directors** think about system architecture and team coordination

When you're a director, you naturally:
- Delegate work to specialists (agents with personas)
- Think about service boundaries and integration
- Prioritize based on business goals, not technical elegance
- Balance speed vs quality vs cost

This reframe forces participants out of "write all the code myself" mode into "orchestrate a team to build systems" mode—which is exactly the multi-agent mindset.

### Learning Through Pressure

**Why quarterly demos?**

Because pressure reveals truth:

- **Without demos:** You plan forever, never ship
- **With demos:** You prioritize ruthlessly, learn what works

The demo pressure forces:
- ✅ Focus on delivery, not perfection
- ✅ Iterative refinement (make it work, then make it good)
- ✅ Clear communication (can you explain what you built?)
- ✅ Real decisions (tradeoffs matter when time is limited)

### Learning Through Competition (Friendly)

**Why VP promotion scoring?**

Because friendly competition drives engagement:

- **Metrics matter:** Delivery score, tech debt, API cost efficiency
- **No sabotage:** Everyone can succeed (not zero-sum)
- **Motivation:** Natural desire to do well, show off cool techniques
- **Reflection:** Scoring forces you to think about tradeoffs

But it's still collaborative—you're all trying to make SkywardAI succeed.

## FAQ

**Q: Do I need to be an expert in AI agents to participate?**
A: No! You should be comfortable using Claude or similar tools, but multi-agent orchestration is what you'll learn during the game.

**Q: What if I'm not familiar with airline domain?**
A: Perfect! The character sheets teach you the business domain so you understand what you're building and why. That's part of the learning.

**Q: Do I need to know specific technologies (React, Python, etc.)?**
A: No specific tech stack is required. You'll decide with your agent teams in Q1 what technologies to use. The game is tech-stack agnostic.

**Q: What if I fall behind in a quarter?**
A: That's fine! Demo what you have, learn from it, improve next quarter. The game is designed for learning, not perfection.

**Q: Can I collaborate with other directors?**
A: Yes, but it's optional. Each director works independently with mock data. Real integration is bonus enrichment in Q4 if time allows.

**Q: What if my agents produce broken code?**
A: That's a learning opportunity! Use other agents to debug, or try different prompting strategies. This is a safe space to experiment.

**Q: Will I learn management skills, or just technical skills?**
A: Primarily technical (multi-agent orchestration), but you'll naturally gain insights into management thinking as a bonus from the director role-play.
